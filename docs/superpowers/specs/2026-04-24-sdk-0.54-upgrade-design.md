# Cosmos SDK 0.54 Upgrade — Design

**Branch:** `sdk54` (created by the user before the implementation pass starts)
**Target:** Cosmos SDK v0.53.7 → v0.54.2, plus coordinated bumps across CometBFT, ibc-go (v10 → v11), wasmd / wasmvm, interchain-security, ibc-apps, and the Stride-Labs vesting fork.

## Goal and non-goals

### Goal

Upgrade Stride's dependency stack from SDK 0.53.7 → SDK 0.54.2 and adapt the codebase to the corresponding API changes. Work is organized into eight per-dependency commits so each can be reviewed independently.

### Non-goals (explicit exclusions)

- **Adopting any new SDK 0.54 features.** No OpenTelemetry telemetry replacement, centralized authority via `AuthorityParams`, BlockSTM, LibP2P / AdaptiveSync, `x/protocolpool`, or PoA module. Minimal route only.
- **Stride module path bump (`github.com/Stride-Labs/stride/v32 → v33`).** Not referenced in any commit. The user handles this manually closer to deploy; including it in intermediate diffs would clog every PR.
- **Upgrade handler for the v33 release.** Not in scope. State migrations (ibc-go v11 `ParamSubspace` cleanup, ICS v8, the four `cosmossdk.io/x/*` module-path migrations) will be handled by a future upgrade handler authored separately as part of the release packaging.
- **Build / compile / test verification.** The repo is expected to be in a broken build state across all intermediate commits and may still be broken at the end. No test work in scope.
- **Proto regeneration, changelog, release PR.** Deferred or handled separately.
- **POA module migration.** The previous SDK 53 design considered POA as a possible ICS replacement; that's no longer needed since upstream ICS shipped SDK 0.54 + ibc-go v11 support to main on 2026-04-22. Stride continues using ICS.

## Commit plan

Eight commits on `sdk54`, bottom-up along the dependency tree. Each commit combines the `go.mod` bump and the code adaptations for that dependency. Repo allowed to be non-building between commits.

### Commit 1a — `bump cosmos-sdk to v0.54.2 and cometbft to v0.39.1`

**go.mod:**

- `cosmos-sdk` v0.53.7 → v0.54.2.
- `cometbft` v0.38.21 → v0.39.1.
- `cosmossdk.io/{api, core, errors, client/v2, tools/confix}` bumped to versions compatible with SDK 0.54 per the version matrix.
- Leave `cosmossdk.io/log`, `cosmossdk.io/store`, and the four `cosmossdk.io/x/*` modules in place — they get rewritten in 1b.
- Re-evaluate any existing replace directives that pin SDK ecosystem deps; remove if no longer needed against 0.54.

**Code changes (strictly required by SDK 0.54):**

1. **`x/gov` keeper signature.** `govkeeper.NewKeeper` no longer takes `app.StakingKeeper`. Drop that arg and append `govkeeper.NewDefaultCalculateVoteResultsAndVotingPower(app.StakingKeeper)` as the last param. Order shifts so bank precedes distr directly.
2. **`GovHooks.AfterProposalSubmission` signature.** Add `proposerAddr sdk.AccAddress` as the third parameter to any custom impl Stride has.
3. **`x/bank` end-blocker registration (mandatory).** Add `banktypes.ModuleName` (first) to `app.ModuleManager.SetOrderEndBlockers(...)`. Required even without BlockSTM.
4. **`NodeService` registration.** `RegisterNodeService` now takes a 4th arg, an `EarliestVersion` callback: `func() int64 { return app.CommitMultiStore().EarliestVersion() }`.

The SDK's `x/epochs.NewAppModule` keeper-by-pointer change does NOT apply — Stride uses its own `x/epochs` module, not the SDK's. (Same exclusion held for the SDK 53 upgrade.)

All other 0.54 changes (BlockSTM, OpenTelemetry, AuthorityParams, LibP2P, AdaptiveSync, PoA, x/protocolpool) are opt-in and excluded.

## Known non-building states

- **Intermediate commits:** the repo will not compile between commits. Every dependency bump breaks the ones above it in the dep tree until their own commits land.
- **After commit 6:** expected to build, *assuming* the three Stride-Labs ibc-apps RC tags resolve cleanly and the ICS pseudo-version off main doesn't surface unexpected wiring shifts. A "compilation fixes" follow-up commit is possible but not planned upfront.
- **If the vesting fork tag were missing** (it isn't, but for completeness): commit 6 would be deferred and the repo would remain non-building specifically on the vesting integration.

## Risks

- **`ParamSubspace` removal is wider than the migration doc shows.** ibc-go v11 migration page diffs only `ibckeeper.NewKeeper`, but PR #8476 removed it from *all* keeper constructors. Stride must check transfer, ICA host, ICA controller, and 08-wasm keepers individually.
- **`SetICS4Wrapper` / `SetUnderlyingApplication` interface additions.** NOT mentioned in the published v10 → v11 migration doc — surfaces as compile errors in commit 2b. Every Stride `IBCModule` and every middleware needs them. Easy to miss until late in the pass.
- **Store v2 in-tree path migration.** `cosmossdk.io/store` becomes `github.com/cosmos/cosmos-sdk/store/v2` — both path AND major bump simultaneously. Sweep must catch every direct usage.
- **wasmvm `/v2 → /v3` path bump** in lockstep with wasmd. wasmvm direct imports in Stride are limited but exist (e.g. types in wasm wiring) — sweep must catch them all.
- **ICS pseudo-version off main.** If the Go module proxy ever evicts the chosen commit, a fresh pick from ICS main is needed. ICS main may also shift APIs after our pin without notice.
- **ibc-apps Stride-Labs RC tags may evolve.** If new `-stride-rc1` (or later) tags drop mid-implementation, decide whether to re-pin or freeze on rc0.
- **CometBFT 0.39 surprises.** SDK 0.54's `BaseApp` adapts the integration internally, but if Stride has any direct cometbft imports (RPC clients in tests/scripts, custom block-handling), those may break. Likely a small surface — worth a grep pass at impl.
- **Stride custom IBC modules.** ~10 files implement `IBCModule` / `Middleware`. Each needs the two new interface methods added — formulaic but easy to miss one.
- **No compile checkpoints.** Inherits the same accepted risk as the SDK 53 plan: a mistake in commit 1a may only surface during commit 5 or 6.

## Ordering rationale

The SDK sits at the root of the dependency graph — every other package in the matrix imports it directly or transitively. Upgrading SDK first means each subsequent commit can adapt its consumer against the already-updated SDK APIs rather than fighting two moving targets at once.

Within the SDK upgrade itself, version + wiring (1a) precedes the path migrations (1b). The version bump is what *forces* the path migrations to be valid — splitting them gives reviewers a clean conceptual commit (the wiring API changes mandated by 0.54) followed by a clean mechanical commit (every `cosmossdk.io/log → log/v2`, `cosmossdk.io/x/* → cosmos-sdk/x/*`, `store → store/v2` rewrite). This mirrors the 2a/2b pattern the SDK 53 plan applied to ibc-go.

ibc-go (2a/2b) follows the same mechanical-then-conceptual split: path rewrite first, API adaptation second. No "remove capability" sub-commit this round — capability was already removed in the SDK 53 upgrade.

The siblings on top of ibc-go + SDK come next: ibc-apps (3) → wasmd (4) → ICS (5). Order chosen by expected code impact and to keep ICS — the most volatile, since it's a pseudo-version off main — last among that group.

Vesting (6) is bumped last. The tag already exists, so it's a single go.mod replace-directive edit with no code changes expected.
