# Cosmos SDK 0.53 Upgrade — Design

**Branch:** `sdk53`
**Target:** Cosmos SDK v0.50.14 → v0.53.7, plus coordinated bumps across the IBC / CosmWasm / ICS stack.

## Goal and non-goals

### Goal

Upgrade Stride's dependency stack from SDK 0.50 → SDK 0.53 and adapt the codebase to the corresponding API changes. Work is organized into a small series of per-dependency commits so each can be reviewed independently.

### Non-goals (explicit exclusions)

- **SDK 0.54 / cometbft 0.39 / ibc-go v11 / `cosmossdk.io/log/v2` / `cosmossdk.io/client/v2 v2.11` / `cosmossdk.io/store/v2`.** The 0.54 line is out of scope. A future upgrade will need to pair SDK 0.54 with the POA module to replace ICS, since no ICS version exists for SDK 0.54.
- **POA module migration / ICS removal.** Separate future effort. This upgrade keeps ICS.
- **IBC v2 Transfer stack.** Not wired. Stride does not need it today.
- **Opt-in SDK 0.53 features:** unordered transactions, `x/protocolpool`, SDK's `x/epochs` (Stride has its own), and the `x/distribution` handler removal that only applies when adopting `x/protocolpool`.
- **Stride module path bump (`github.com/Stride-Labs/stride/v31 → v32`).** Not referenced in any commit in this plan. The user handles this manually closer to deploy; including it here would clog every intermediate diff.
- **Build / compile / test verification.** The repo is expected to be in a broken build state across all intermediate commits and may still be broken at the end (see "Known non-building states" below). No test work in scope.
- **Proto regeneration, changelog, upgrade handler, release PR.** All deferred or handled separately.

## Commit plan

Eight commits on `sdk53`, bottom-up along the dependency tree. Approach 2: each commit combines the `go.mod` bump and the code adaptations for that dependency. Repo is allowed to be non-building between commits.

### Commit 1 — `bump cosmos-sdk to v0.53.7`

**go.mod:**
- Bump `cosmos-sdk` to v0.53.7.
- Bump all `cosmossdk.io/*` submodules to latest versions compatible with SDK 0.53 (log on v1.x, client/v2 latest beta, store on v1.x, api, core, math, collections, errors, x/tx, x/upgrade, x/feegrant, x/evidence, x/circuit, depinject, tools/confix).
- `cometbft` stays at v0.38.21.
- Re-evaluate the existing `cosmossdk.io/core => v0.11.0` replace directive — drop it if SDK 0.53 no longer needs the `comet.BlockInfo` workaround.

**Code changes, strictly from `cosmos-sdk/v0.53.7/UPGRADING.md` 0.50 → 0.53 section:**

1. **`x/auth` PreBlocker registration (REQUIRED).** Add `authtypes.ModuleName` to `app.ModuleManager.SetOrderPreBlockers(...)` after `upgradetypes.ModuleName`.
2. **`x/mint` keeper constructor (conditional).** `InflationCalculationFn` is deprecated. Pass `nil`; if custom inflation is used, wrap with `mintkeeper.WithMintFn(mintkeeper.DefaultMintFn(customFn))`. Stride's custom `x/mint` module may not use the SDK keeper directly — inspect during implementation.
3. **CLI `testnet init-files` flag (conditional).** `--v` → `--validator-count` if any scripts/docs reference the old flag.

All other 0.53 changes (unordered transactions, `x/protocolpool`, `x/distribution` handler removal, SDK's `x/epochs`) are opt-in and explicitly excluded.

### Commit 2a — `bump ibc-go to v10 and rewrite import paths`

**go.mod:**
- `github.com/cosmos/ibc-go/v8 → /v10 v10.5.1` (path change).
- `github.com/cosmos/ibc-go/modules/light-clients/08-wasm → .../08-wasm/v10 v10.5.0` (path change).
- `github.com/cosmos/ibc-go/modules/capability` retained until commit 2b.

**Code changes:**
- Mechanical rewrite `ibc-go/v8/` → `ibc-go/v10/` throughout `app/app.go` and every custom `x/*` module that imports ibc-go.
- Mechanical rewrite `08-wasm/` imports to the `/v10` path.
- Fix any trivial rename / relocation breakages that surface from the path bump.

Intent: large line count but low reasoning load. Reviewable as "mostly mechanical".

### Commit 2b — `remove capability module`

**go.mod:**
- Remove `github.com/cosmos/ibc-go/modules/capability`.

**Code changes:**
- Delete the `CapabilityKeeper` field and its instantiation from app.
- Delete `ScopedIBCKeeper`, `ScopedTransferKeeper`, `ScopedICAHostKeeper`, `ScopedICAControllerKeeper` (and any other scoped keepers) — fields and all `ScopeToModule(...)` calls.
- Drop `capabilitytypes.StoreKey` from `NewKVStoreKeys` and remove the capability in-memory store key.
- Remove `capability.NewAppModule(...)` from the module manager and all `SetOrderBeginBlockers`, `SetOrderEndBlockers`, `SetOrderInitGenesis`, `SetOrderExportGenesis` entries.
- Remove any capability-related module account permissions.

Standalone commit so reviewers see the most impactful conceptual change in isolation.

### Commit 2c — `adapt app wiring and custom IBC modules to ibc-go v10 APIs`

**Code changes, strictly from the v8.1 → v10 migration guide:**

- **Core IBC keeper:** `keys[ibcexported.StoreKey]` → `runtime.NewKVStoreService(keys[ibcexported.StoreKey])`; remove `StakingKeeper` and `scopedIBCKeeper` params.
- **Transfer keeper:** store-service wrap; `app.MsgServiceRouter()` replaces `app.IBCKeeper.PortKeeper`; drop `scopedTransferKeeper`; after keeper construction, wrap ICS4 by calling `app.TransferKeeper.WithICS4Wrapper(<top-of-stack>)` where `<top-of-stack>` is the outermost middleware wrapping Stride's transfer module (i.e., the rate-limit / PFM / ibc-hooks chain root).
- **ICA Host keeper:** store-service wrap; `app.IBCKeeper.ChannelKeeper` replaces `PortKeeper`; drop `scopedICAHostKeeper`; pass `app.GRPCQueryRouter()` directly (remove the separate `WithQueryRouter()` call).
- **ICA Controller keeper / stack:** remove the `noAuthzModule` empty `IBCModule` wrapper.
- **Light client module registration (new architecture):** register `07-tendermint` and `08-wasm` as separate modules via `clientKeeper.AddRoute(moduleName, &lightClientModule)`; pass the tendermint light client module instance to `ibctm.NewAppModule(tmLightClientModule)`; obtain the store provider from `app.IBCKeeper.ClientKeeper.GetStoreProvider()`.
- **08-wasm specifics:**
  - `ibcwasmtypes.QueryPlugins` → `ibcwasmkeeper.QueryPlugins`.
  - `ibcwasmtypes.AcceptListStargateQuerier(...)` → `ibcwasmkeeper.AcceptListStargateQuerier(..., app.GRPCQueryRouter())`.
  - Use `app.WasmClientKeeper.InitializePinnedCodes()` instead of the module-level function.
- **Governance router:** remove the legacy 02-client proposal handler.
- **Custom IBC modules:** sweep every `IBCModule` implementation in Stride's custom modules (`x/stakeibc`, `x/autopilot`, `x/icaoracle`, `x/icacallbacks`, `x/interchainquery`, anywhere else implementing the interface). Update callback signatures for `OnChanOpenInit/Try/Ack/Confirm`, `OnChanCloseInit/Confirm`, `OnRecvPacket`, `OnAcknowledgementPacket`, `OnTimeoutPacket` — capability references removed, signature shape changed.
- **Middleware stack:** callbacks middleware wraps the base transfer module directly; PFM sits above callbacks in the stack; no fee middleware anywhere. Stride does not use 29-fee (nothing to remove) and does not use ibc-go's `apps/callbacks` (Stride's `x/icacallbacks` is unrelated).

## Known non-building states

- **Intermediate commits:** the repo will not compile between commits. Every dependency bump breaks the ones above it in the dep tree until their own commits land.
- **After commit 6:** expected to build, assuming no residual surprises (custom module API shifts, transitive version conflicts). A "compilation fixes" follow-up commit is possible but not planned up front.
- **If the vesting fork tag is not ready:** commit 6 is skipped and the repo remains non-building specifically on the vesting integration. All other commits still land.

## Risks

- **Large commit 2 variants.** Even split into 2a/2b/2c, each piece still touches many files. Capability removal (2b) and wiring updates (2c) have the highest surprise potential.
- **ibc-hooks v10.0.0 stale pin.** Its go.mod still pins SDK v0.50.14. Go MVS should lift this to 0.53 transitively, but if the module uses an API removed in 0.53 we may need a `replace` pointing at a newer commit on their main branch, or to temporarily vendor.
- **ICS pseudo-version availability.** If the Go module proxy ever evicts `v7.0.0-20250408210344-06e0dc6bf6d6`, a fresh commit pick from ICS `main` is needed.
- **Stride custom IBC modules.** Multiple implementations of `IBCModule` in custom `x/*` — each needs per-file callback signature inspection. Scope is hard to estimate until the implementation pass.
- **Stride custom `x/mint`.** May or may not collide with SDK 0.53's `WithMintFn` injection. Commit 1 absorbs a small wrapper if needed.
- **No compile checkpoints.** A mistake in commit 1 might only surface during the commit 5 or 6 sweep. Accepted per deferred-testing constraint.

## Ordering rationale

The SDK sits at the root of the dependency graph — every other package in the matrix imports it directly or transitively. Upgrading SDK first means each subsequent commit can adapt its consumer against the already-updated SDK APIs rather than fighting two moving targets at once. So the order is: SDK (1) → ibc-go (2a/2b/2c) → ibc-apps / wasmd / ICS (3, 4, 5 — siblings that all sit on top of ibc-go + SDK, ordered by expected code impact) → vesting (6, bumped last because its new fork tag depends on the user publishing it).

Within ibc-go, the three sub-commits (2a/2b/2c) go: mechanical path rewrite → capability removal → remaining wiring and callback signature updates. Capability removal is isolated in 2b so reviewers can focus on the single most impactful conceptual change without it drowning in path-rewrite noise.
